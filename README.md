# CS 300 Portfolio Reflection

## Project One: Data Structures Analysis

In Project One, I analyzed the run-time and memory usage of different data structures. The goal was to compare options such as vectors, hash tables, and binary search trees to determine which structure would be most efficient for storing, searching, and printing course information.

## Project Two: Course Planner Program

In Project Two, I created a working C++ program that reads course information, sorts the courses in alphanumeric order, and prints the course list for the Computer Science program. This project helped me understand how data structures can be used to organize and retrieve information efficiently.

## Reflection

The main problem I was solving in these projects was how to organize and access course data in an efficient way. Data structures are important because they affect how quickly a program can search, sort, insert, and display information. Choosing the right data structure can make a program faster, easier to maintain, and better suited for real-world use.

I approached the problem by first reviewing the requirements and then considering which data structure would work best for the task. I learned that each structure has strengths and weaknesses. For example, vectors are simple and easy to use, hash tables are useful for fast lookups, and binary search trees are helpful for keeping data organized in sorted order.

One roadblock I encountered was making sure the program correctly read the course data and printed it in the required order. I overcame this by testing the program, reviewing errors carefully, and breaking the problem into smaller steps. This helped me understand where issues were happening and how to fix them.

Working on these projects expanded the way I approach software design. I now think more carefully about planning before coding, choosing the right structure for the problem, and writing code that can be understood by another programmer. I also learned the importance of documenting my work so that someone reviewing my GitHub portfolio can understand what the project does and why I made certain design choices.

This project also helped me improve the way I write programs that are maintainable, readable, and adaptable. I focused on organizing the code into functions, using clear names, and keeping the logic easy to follow. These habits will help me continue improving as a programmer and prepare me for future projects.



CS230 GAME ROOM

Briefly summarize The Gaming Room client and their software requirements. Who was the client? What type of software did they want you to design?
The client, The Gaming Room, is a company that wanted to expand their existing game, Draw It or Lose It, into a web-based application that could support multiple teams and players across different platforms. Their main requirement was to create a scalable, secure, and cross-platform system that ensures only one instance of the game runs at a time while maintaining unique team and player names. The software needed to be user-friendly, reliable, and capable of handling multiple concurrent users.

What did you do particularly well in developing this documentation?
I did particularly well in organizing the design document in a clear and structured way. I made sure each section—such as system architecture, platform evaluation, and requirements—was easy to follow and logically connected. I also did a strong job comparing different operating platforms and justifying my final recommendation based on performance, scalability, and cost considerations.

What about the process of working through a design document did you find helpful when developing the code?
Working through the design document before coding was extremely helpful because it gave me a clear roadmap for development. It allowed me to think through system requirements, constraints, and architecture decisions ahead of time, which reduced confusion later. Having a structured plan made the coding process more efficient and minimized potential rework.

If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?
If I could revise one part of my work, I would improve the system architecture diagrams and add more detail. While they were functional, I think they could be clearer and more visually descriptive. I would also expand on certain technical justifications to better explain why specific technologies or approaches were chosen.


How did you interpret the user’s needs and implement them into your software design? Why is it so important to consider the user’s needs when designing?
I interpreted the user’s needs by focusing on the client’s goals—scalability, usability, and cross-platform compatibility—and translating them into technical requirements. For example, ensuring unique user and team names and supporting multiple users influenced the system design and database structure. Considering user needs is critical because it ensures the final product is practical, usable, and aligned with what the client actually wants, rather than just what is technically possible.

How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?
My approach to designing software involved analyzing requirements first, then evaluating different solutions before selecting the best option. I focused on scalability, maintainability, and performance when making decisions. In the future, I would continue using techniques such as requirement analysis, system modeling (like UML diagrams), and platform comparison. I would also incorporate more iterative design and user feedback earlier in the process to improve the overall quality of the software.
